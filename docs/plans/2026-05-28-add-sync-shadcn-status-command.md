Findings:
- `sync-shadcn` had `review` as a full staleness audit, but no quick status
  command for deciding what to do next.
- Deferred items need to come from durable state first, then plan questions and
  unimplemented/deferred slices.

Decisions and tradeoffs:
- `status` is read-only and does not write artifacts. Full evidence belongs in
  `sync-shadcn review`.
- `status` may fetch upstream for freshness, but if fetch fails it reports
  freshness as unverified instead of mutating state or pretending confidence.
- Settled exclusions and preserved forks are shown as context, not questions.
